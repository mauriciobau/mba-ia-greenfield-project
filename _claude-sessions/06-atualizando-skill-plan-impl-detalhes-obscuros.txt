     Plan: Improve SKILL.md Validation вҖ” Catch Unmapped Consequences of 
     Requirements

     Context

     During Phase 02 planning (auth), the validation step failed to catch that
     channel handles derived from email prefixes can collide when two users
     share the same prefix (e.g., john@gmail.com and john@outlook.com вҶ’ both
     produce handle john). The user had to point this out manually.

     The root issue is broader than derived values: functional requirements have
      implicit consequences that aren't mapped in any document. The existing 4
     validation categories compare documents statically (contradictions,
     vagueness, missing choices, dependency gaps). None of them force the agent
     to mentally simulate a requirement's execution end-to-end and check if all
     consequences are addressed.

     What to change

     File: .claude/skills/plan-phase/SKILL.md (lines 39вҖ“43)

     1. Add a 5th validation category: "Unmapped consequences of functional 
     requirements"

     Insert after "Dependency gaps" (line 41) and before "If any issues are
     found" (line 43).

     Key design: This is NOT a checklist of specific scenarios. It's a
     systematic process вҖ” 5 guiding questions the agent must ask for each 
     functional requirement in the phase:

     **Unmapped consequences of functional requirements:**

     Unlike the categories above вҖ” which compare documents against each other
     statically вҖ” this one requires mentally simulating each functional
     requirement's execution and checking whether all consequences are
     addressed. For every capability in the phase, walk through the operation
     step by step and ask:

     - **What are the inputs, and what are their edge cases?** Consider
     realistic variance in the data that triggers the operation (e.g., "channel
     handle from email prefix" вҖ” what if two users share the same prefix? What
     if the prefix is empty, one character, or 200 characters? What if it
     contains only special characters that get stripped?).
     - **What does this operation produce, and can those outputs collide, 
     overflow, or violate constraints?** Trace each derived or generated value
     to where it is stored or used. Check whether uniqueness constraints, length
      limits, format requirements, or downstream consumers can be violated by
     legitimate inputs (e.g., a derived handle that is unique in the spec but
     not unique in practice given the derivation rule).
     - **What happens to related entities and side effects?** When the operation
      creates, modifies, or deletes data, trace the impact on every entity it
     touches directly or transitively. Ask whether cascading effects, ordering
     dependencies, or state transitions in related entities are specified (e.g.,
      "delete user" вҖ” are their videos, comments, and upload jobs handled?
     "change email" вҖ” does the channel handle update too, or is it frozen?).
     - **What happens under concurrency or repeated execution?** Consider
     whether two users or two requests performing the same operation
     simultaneously could produce race conditions, duplicate records, or
     inconsistent states not addressed by the specs (e.g., two simultaneous
     registrations with the same email, concurrent refresh token rotations).
     - **Is the failure path specified?** For each step that can fail, check
     whether the specs define what happens вҖ” rollback behavior, error response,
     user-facing message, or cleanup of partial state (e.g., "registration
     creates user + channel" вҖ” what if channel creation fails after the user row
      is inserted? Is it transactional?).

     The goal is not to enumerate every possible edge case exhaustively, but to
     simulate realistic execution paths and flag consequences that no document
     currently addresses. Report each finding the same way as other validation
     issues: quote the requirement, describe the unmapped consequence, and ask
     the user to decide.

     2. Update the grouping instruction

     In the "If any issues are found" paragraph, update the grouping list:

     Before: (inconsistency, ambiguity, missing decision, dependency gap)
     After: (inconsistency, ambiguity, missing decision, dependency gap, 
     unmapped consequence)

     Why this approach

     - Process over checklist: 5 guiding questions that work for any
     requirement, not a finite list of known scenarios (derived values,
     normalization, etc.)
     - Distinct cognitive mode: Existing categories compare docs statically.
     This one forces simulation: "if I execute this requirement, what happens?"
     - Would have caught the handle collision: Walking through "channel handle
     from email prefix" вҶ’ question 1 flags "what if two users share the prefix?"
      вҶ’ question 2 traces the handle to a unique constraint and finds no
     collision strategy вҶ’ reported as unmapped consequence
     - Catches future issues too: Concurrency, failure paths, cascade effects вҖ”
     all classes of unmapped consequences that would otherwise slip through

     2. Grouping list includes all 5 types
     3. No overlap with existing bullets (especially "Missing edge cases" under
     Ambiguities вҖ” that one is about vague scope, this is about precise specs
     with unaddressed consequences)
     4. Introductory sentence makes the distinction clear
